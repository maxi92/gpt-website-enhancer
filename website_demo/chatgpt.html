在 MyBatis 中，@Intercepts 和 @Signature 注解用于定义自定义拦截器的拦截范围。

1. @Intercepts 和 @Signature 的基本含义

  • @Intercepts：标注拦截器类，可以包含一个或多个 @Signature。
  • @Signature：标注具体拦截目标，定义要拦截的接口、方法及其参数。

2. @Signature 的字段说明

  • type：指定要拦截的接口类型，比如 Executor、ParameterHandler、ResultSetHandler 或 StatementHandler。
  • method：指定接口中要拦截的方法名。
  • args：指定被拦截方法的参数类型数组，用于准确定位重载方法。

3. 常见拦截接口

MyBatis 中可以被拦截的接口主要有以下几种：

接口作用常见方法
Executor负责执行增删改查操作的核心接口。query（查询）、update（更新）、commit（提交事务）、rollback（回滚事务）等。
ParameterHandler负责对 SQL 语句的参数处理，将参数设置到 JDBC Statement 中。setParameters
ResultSetHandler负责处理查询结果集，将 JDBC 返回的数据映射到 Java 对象。handleResultSets
StatementHandler负责管理和处理 SQL 语句的生成与执行，封装了 JDBC Statement 的操作。prepare、parameterize、batch 等。

4. 示例分析

以下代码片段拦截了 Executor.query 方法：

java
@Intercepts({
    @Signature(type = Executor.class, method = "query",
        args = {MappedStatement.class, Object.class, RowBounds.class, ResultHandler.class})
})
public class QueryInterceptor implements Interceptor {
    @Override
    public Object intercept(Invocation invocation) throws Throwable {
        // 拦截逻辑
        System.out.println("Before query");
        Object result = invocation.proceed();
        System.out.println("After query");
        return result;
    }
}
  • 拦截内容：
    • 接口：Executor
    • 方法：query
    • 参数：MappedStatement（SQL 的映射信息）、Object（参数对象）、RowBounds（分页信息）、ResultHandler（结果处理器）。

5. 不同的 type 和 method 的区别

通过 type 和 method 的不同组合，可以拦截 MyBatis 执行流程的不同阶段。例如：

  1. 拦截参数处理（ParameterHandler.setParameters）
    在 SQL 参数被设置到 JDBC Statement 之前进行拦截，适合做参数加密、默认值补充等操作。

    java
    @Intercepts({
    @Signature(type = ParameterHandler.class, method = "setParameters", args = {PreparedStatement.class})
})

  2. 拦截 SQL 生成（StatementHandler.prepare）
    在 SQL 语句准备好之前进行拦截，可以对 SQL 语句进行修改（如添加动态表名、数据脱敏）。

    java
    @Intercepts({
    @Signature(type = StatementHandler.class, method = "prepare", args = {Connection.class, Integer.class})
})

  3. 拦截查询结果处理（ResultSetHandler.handleResultSets）
    在查询结果集被映射为 Java 对象之前进行拦截，可以做数据脱敏或结果过滤。

    java
    @Intercepts({
    @Signature(type = ResultSetHandler.class, method = "handleResultSets", args = {Statement.class})
})

  4. 拦截增删改查（Executor.update 和 Executor.query）
    可以记录日志、统计执行时间，或实现动态权限控制。

    java
    @Intercepts({
    @Signature(type = Executor.class, method = "update", args = {MappedStatement.class, Object.class})
}),
@Intercepts({
    @Signature(type = Executor.class, method = "query", args = {MappedStatement.class, Object.class, RowBounds.class, ResultHandler.class})
})

6. 执行流程和拦截点

MyBatis 执行一个 SQL 操作的流程可以概括为：

  1. 创建 MappedStatement，将 SQL 和参数绑定。
  2. 通过 ParameterHandler 设置参数。
  3. 使用 StatementHandler 生成 SQL，并与数据库交互。
  4. 通过 ResultSetHandler 处理查询结果。
  5. 通过 Executor 管理事务、缓存和数据库操作。

不同 type 和 method 对应的拦截点就嵌入在这些步骤中。

7. 拦截器应用场景

  • 日志记录：记录 SQL 执行时间、参数和结果。
  • 权限控制：在查询或更新操作之前验证用户权限。
  • 动态 SQL：修改 SQL，例如多租户场景中动态添加租户过滤条件。
  • 数据脱敏：拦截查询结果，对敏感字段进行脱敏处理。

    @Override

    String originalSql = boundSql.getSql();
    String modifiedSql = modifySql(originalSql);

    BoundSql newBoundSql = new BoundSql(
        ms.getConfiguration(),
        modifiedSql,
        boundSql.getParameterMappings(),
        boundSql.getParameterObject()
    );

    MappedStatement newMs = new MappedStatement.Builder(
        ms.getConfiguration(),
        ms.getId(),
        new BoundSqlSource(newBoundSql),
        ms.getSqlCommandType()
    ).resultMaps(ms.getResultMaps()).build();

    args[0] = newMs;

    // 继续后续拦截器的执行
    return invocation.proceed();
}

// 自定义条件检查方法
private boolean shouldIntercept(Invocation invocation) {
    // 检查条件，例如根据 MappedStatement 的 ID 或方法名判断
    MappedStatement ms = (MappedStatement) invocation.getArgs()[0];
    return ms.getId().contains("TargetKeyword");
}

// 示例的 SQL 修改方法
private String modifySql(String originalSql) {
    return originalSql + " /* Modified by Interceptor */";
}

关键点讲解

  1. invocation.proceed() 的作用：

    • 如果不作任何操作，直接调用 proceed()，MyBatis 会继续调用拦截器链中的下一个拦截器。
    • 如果没有剩余拦截器，proceed() 会执行原始方法。

  2. 条件判断：

    • 在 shouldIntercept 方法中，开发者可以根据业务逻辑动态决定是否需要拦截。
    • 常见条件包括 SQL 的类型、MappedStatement 的 ID、参数内容等。

  3. 确保不破坏链式调用：

    • 即使当前拦截器不执行操作，也应调用 proceed()，确保其他拦截器仍然有机会拦截。

  4. 多个拦截器的执行顺序：

    • 拦截器是按添加的顺序依次执行的。
    • 后续拦截器可以看到前面拦截器的修改结果，因此如果一个拦截器修改了 MappedStatement 或参数，后续拦截器会接收修改后的内容。

示例场景

场景：只有在查询语句包含某些关键字时才拦截

java
private boolean shouldIntercept(Invocation invocation) {
    MappedStatement ms = (MappedStatement) invocation.getArgs()[0];
    BoundSql boundSql = ms.getBoundSql(invocation.getArgs()[1]);
    String sql = boundSql.getSql();
    // 仅在 SQL 包含关键字 "LIMIT" 时进行拦截
    return sql.contains("LIMIT");
}

注意事项

  1. 拦截器的顺序：

    • 多个拦截器的顺序在 MyBatis 配置文件或注册时决定。需要确保顺序符合业务逻辑。

  2. 减少副作用：

    • 一个拦截器的修改可能影响后续拦截器的行为，因此需确保拦截器的作用范围清晰。

  3. 性能问题：

    • 每个拦截器都会执行一遍，因此条件判断应尽量高效，以减少不必要的性能消耗。

通过上述方式，你可以灵活地控制 MyBatis 的拦截器行为，避免某些拦截器在不必要时干扰其他拦截器的工作。
